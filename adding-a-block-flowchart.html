1 Schema Definition

Define the Block Type & Settings

Every block starts as a type entry in the card's section schema. The block's type must be globally unique across Shopify and follow the naming convention _card_product_card__ + block name. Settings define what the merchant configures in the Theme Editor.

// Inside the card section schema's "blocks" array // Each block type has a name, type, and settings array { "name": "Testimonial", "type": "_card_product_card__testimonial", "settings": [ // --- Testimonial-specific settings --- { "type": "inline_richtext", "id": "author", "label": "Author name", "default": "[product.vendor]" // <-- dynamic text! }, { "type": "richtext", "id": "content", "label": "Review text", "default": "<p>Great product!</p>" }, { "type": "range", "id": "rating", "label": "Rating (1-5)", "min": 1, "max": 5, "step": 1, "default": 5 }, { "type": "image_picker", "id": "avatar", "label": "Author photo (optional)" }, // --- Star color overrides (fall back to global theme settings) --- { "type": "color", "alpha": true, // <-- always use alpha: true for colors "id": "star_active_color", "label": "Star active color", "info": "Overrides global settings.color__review_stars" }, { "type": "color", "alpha": true, "id": "star_base_color", "label": "Star inactive color", "info": "Overrides global settings.color__review_stars_base" }, // --- Standard settings (every block should include) --- { "type": "color", "alpha": true, "id": "text_color", "label": "Text color" }, { "type": "color", "alpha": true, "id": "background_color", "label": "Background color" }, { "type": "text", "id": "css_class", "label": "CSS class" }, { "type": "textarea", "id": "custom_css", "label": "Custom CSS" }, { "type": "select", "id": "show_conditionally", "label": "Show conditionally", "default": "always", "options": [ { "value": "always", "label": "Always" }, { "value": "with_subscriptions", "label": "Subscriptions only" }, { "value": "no_subscriptions", "label": "No subscriptions" } // ... more options ] } ] }
Key convention: "default": "[product.vendor]" uses bracket notation to indicate a dynamic text value. This means the field can pull live product data — not just static text. The brackets are resolved differently depending on the render path (server-side Liquid vs client-side JS).
Defined in the section schema JSON (inside card_product_card.liquid or metaobject schema)
2 Prop Drilling — _render_blocks

Register the Block in the Render Dispatcher

The _render_blocks.liquid snippet is the central dispatcher. It iterates block_order, checks each block's type, and renders the matching .static.liquid file. You must add a new if block for the testimonial type.

Every block receives the same prop signature — this is the "prop drilling" pattern. The parent passes everything; the child uses what it needs.

{%- liquid comment -%} Add this block alongside the other type checks in card_product_card._render_blocks.liquid {%- endcomment -%} if block.type == '_card_product_card__testimonial' render 'card_product_card.testimonial.static', section_id: section_id, block_id: block_id, style_id: style_id, selector: selector, product: product, variant: variant, section: section, block: block, preorder: preorder, preorder_date: preorder_date, preorder_quantity: preorder_quantity, price: price, compare_at_price: compare_at_price, is_bundle: is_bundle, is_pdp: is_pdp, preload_images: preload_images endif
PropTypeUsed by testimonial?
style_idstringYes — for data-style-id on the root element
blockobjectYes — contains .settings (author, content, rating, etc.)
productproductYes — needed by _input.dynamic_content for bracket resolution
variantvariantYes — needed for [selected_variant.*] dynamic text
section_idstringNo — but always passed for consistency
block_idstringYes — used for show_conditionally editor detection
pricemoneyNot directly, but available if needed
is_bundlebooleanPossibly — for show_conditionally logic
Why pass everything? Liquid render creates an isolated scope — the child snippet can only access variables explicitly passed to it. This is why every block gets the full prop signature, even if it doesn't use all of them. It's not wasteful: unused variables are just ignored.

4 CSS — Block-Type Styles

Add Custom CSS Rules in _styles.liquid

If your block needs type-specific CSS beyond text_color / background_color (which are handled by the base & { } rule), add a conditional block in card_product_card._styles.liquid.

The & selector is a placeholder that gets replaced with the actual scoped selector at compile time.

What you write (with &)
{% if block.type == '_card_product_card__testimonial' %} {% capture additional_css %} & img { object-fit: cover; } & blockquote { font-style: italic; opacity: 0.9; } {% endcapture %} {% endif %} /* Star colors are NOT in _styles.liquid. They're set as inline style= on the element (see Step 3) so the cascade works: block setting (alpha>0) → global CSS var */
Compiled output (scoped)
/* After: replace '&', selector */ /* selector = [data-style-id="abc123"] */ [data-style-id="abc123"] img { object-fit: cover; } [data-style-id="abc123"] blockquote { font-style: italic; opacity: 0.9; }
How the replacement chain works:
css (base: text_color, background_color) + additional_css (block-type-specific) + _settings.custom_css (merchant overrides) → all & replaced with [data-style-id="..."] → strip_newlines → output.

Line 84: {{ css | append: additional_css | append: _settings.custom_css | replace: '&', selector | strip_newlines }}
Star color cascade pattern: Star colors use inline style= on the element (Step 3), not _styles.liquid. This follows the same pattern as the existing rating block:

1. Block setting has alpha > 0? → Use the block-level color directly: --star-active: #f3e008;
2. Block setting is transparent / unset? → Point to the global CSS variable: --star-active: var(--color-review-stars--hex);

The global variables (--color-review-stars--hex, --color-review-stars-base--hex) are defined in _layout.global_colors.liquid from settings.color__review_stars and settings.color__review_stars_base in settings_schema.ts.

Why alpha: true? All color settings must use "alpha": true in the schema. This gives Shopify's color picker a transparency slider, and lets us check .alpha > 0 in Liquid to detect whether the merchant actually set a color vs. left it empty/transparent.

5 Dynamic Text — Bracket Notation

How [product.vendor] Becomes Real Data

The testimonial's author field has a default of [product.vendor]. This bracket notation is resolved by two completely different systems depending on the render path.

Server-side (Static Cards)
// _input.dynamic_content.liquid // Receives: content, keys, product, variant // 1. Find brackets: [product.vendor] // 2. Split by '.': ["product", "vendor"] // 3. Traverse: product["vendor"] // 4. Return: "Nike" {% capture dynamic_author %} {% render '_input.dynamic_content', content: author, keys: 'product, selected_variant', product: product, selected_variant: variant %} {% endcapture %} // Result: "Nike" (plain text in HTML)
Client-side (Dynamic Cards)
// utils.ts → getBracketInputDynamicPluralizedText() // Called via getDynamicText() in Alpine // 1. Read data-author="[product.vendor]" // 2. Parse brackets // 3. Look up in Alpine state: // p_card.product.vendor → "Nike" // 4. Return: "Nike" x-data="{ author_text: getDynamicText( $el.getAttribute('data-author') ) }" x-text="author_text" // Result: "Nike" (reactive, updates on // variant change)
Special resolution types: The bracket system isn't just for text. It handles:
  • Prices: [selected_variant.price] → formatted with money filter (Liquid) or utils.formatMoney() (JS)
  • Images: [product.featured_image] → processed with image_url filter (Liquid) or utils.getImageSrcSet() (JS)
  • Metafields: [product.metafields.custom.review_text] → checks .type and .value automatically
  • Rich text metafields: Auto-applies metafield_tag filter for HTML output
  • Dates: Converted from Day.js format to Liquid date format server-side
  • Fallback chain: [selected_variant.compare_at_price],[product.compare_at_price] → tries first, falls back to second

Common Pitfalls

Things that trip up developers when adding new blocks

Forgetting x-ignore on static
The .static.liquid file should have x-ignore on the root element. Without it, Alpine will try to initialize the element and may conflict with the server-rendered content. Only add Alpine directives (x-data, x-effect) when is_dynamic is true.
Wrong data-style-id source
Static uses {{ style_id }} (passed from _render_blocks).
Dynamic uses {{ block.id | split: "__" | last }}.
Mixing these up means CSS won't match your elements.
Missing & in CSS rules
Every CSS rule in _styles.liquid must start with &. Writing img { } instead of & img { } creates a global rule that affects ALL images on the page, not just your block.
Not handling the blank metaobject case
The .liquid (dynamic) file must include inline <style> output when metaobjects.product_cards[...] == blank. Otherwise the block renders with no styling at all in the fallback path.
Bracket notation with [selected_*]
Fields using [selected_variant.*] need the is_dynamic check. If you use [product.*] only (no variant dependency), the value is static and doesn't need Alpine reactivity in the .static.liquid file.
Prop signature mismatch
Every block in _render_blocks.liquid must pass the exact same prop list. If you miss a prop (like is_bundle), the show_conditionally logic will silently fail — the variable will be nil instead of false.
